Notification-based flows

VC codes

In RP API v3, VC codes are not generated by RP from DTBS, but are returned from the server to RP. VC codes are only relevant for the notification-based flows.

Currently, there is only supported type of VC codes:

  • alphaNumeric4

The returned VC code consists of 4 characters (numbers and upper-case latin letters). It should be shown to the user as received from the server.

The primary motivation for the change was the fact that the previous VC code format was too similar for PIN1 code and caused confusion for some of the users. Also, the inclusion of letters improves the number of possible combinations, thus strengthening it as a security measure.

This functionality is work-in-progress and this section has not been finalized yet.

Delay API call until user has seen verification code (for notification-based flows)

When Smart-ID is used to perform a transaction on the same device it is installed on (browser or RP app) via the notification-based flows, the push notification can arrive quickly and bring the Smart-ID app into the foreground before the user has had a chance to read the verification code presented by the RP.

This is especially pronounced in the Verification Code Choice scenario as the user will not know the correct code when prompted to choose it. The exact behaviour depends on mobile device operating system version, screen state, app permissions and app configuration.

To give the user time to read the verification code, the RP should include a parameter delayApiCall in the authentication and session requests when RP has detected that the user is intending to use the same-device flow.
