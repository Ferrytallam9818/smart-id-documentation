Certificate choice session

This method initiates certificate choice between multiple signing certificates the user may hold on his/her different mobile devices.

Having a correct certificate is needed for giving signatures under *AdES schemes. The method can be ignored if the signature scheme does not mandate presence of certificate in the document structure that is to be signed.

In practice, the user is the one who confirms which device (and therefore signing certificate) she will use for the upcoming signing operation. How this is done depends on which endpoint/flow the RP has chosen:

  • in the dynamic link flow (the dynamic link based endpoint), the user interacts with the dynamic link (QR code, Web2App link, or App2App link) on a certain device, thereby indicating that this is the device that she intends to use.

  • in the notification-based flow (the notification-based endpoint), a certificate (device) choice dialogue is initiated on the user’s devices. The user interacts with one of the devices.

After the certificate has been determined, it becomes available to the RP via the session result endpoint.

In the notification-based flow, if the user has only one account, the certificate is returned to the RP without involving the mobile device at all.

The method accepts QSCD as a certificate level parameter. This is a shortcut marking a certificate of QUALIFIED level which is also QSCD-capable. ADVANCED certificates cannot be QSCD-capable.

Warning. This method initiates a certificate (device) choice dialogue on end user’s devices, so it may not be called without explicit need (i.e. it may be called only as the first step in the signing process).

Certificate choice session endpoints

For the certificate choice requests, the API descriptions practically coincide. The only differences are:

  • Request endpoints

  • Response structure

In below, only the differing parts are described separately.

Dynamic link based certificate choice endpoints
Method URL

POST

BASE/v3/certificatechoice/dynamic-link/etsi/:semantics-identifier

POST

BASE/v3/certificatechoice/dynamic-link/document/:documentnumber
Notification-based certificate choice endpoints
Method URL

POST

BASE/v3/certificatechoice/notification/etsi/:semantics-identifier

POST

BASE/v3/certificatechoice/notification/document/:documentnumber

Error conditions

  • HTTP error code 403 - Relying Party has no permission to issue the request. This may happen when:

    • Relying Party has no permission to invoke operations on accounts with ADVANCED certificates.

    • Relying Party has no permission to use requested capability.

  • HTTP error code 404 - object described in URL was not found, essentially meaning that the user does not have an account in Smart-ID system.

Request parameters

Certificate choice request parameters
Parameter Type Mandatory Description

relyingPartyUUID

string

+

UUID of Relying Party.

relyingPartyName

string

+

RP friendly name, one of those configured for particular RP. Limited to 32 bytes in UTF-8 encoding.

certificateLevel

string

Level of certificate requested. ADVANCED/QUALIFIED/QSCD, defaults to QUALIFIED.

nonce

string

Random string, up to 30 characters. If present, must have at least 1 character.

capabilities

array

Used only when agreed with Smart-ID provider. When omitted request capabilities are derived from certificateLevel parameter.

requestProperties

object

A request properties object as a set of name/value pairs. IP sharing.
Certificate choice request example
{
  "relyingPartyUUID": "1f1bfa89-4f8b-420a-a98e-fb3a161a30bc",
  "relyingPartyName": "DEMO",
  "certificateLevel": "QUALIFIED"
}

Example response

Certificate choice response
{
  "sessionID": "de305d54-75b4-431b-adb2-eb6b9e546014"
}

Response on successful session creation

The response depends on whether the notification-based or dynamic link based endpoint was called.
